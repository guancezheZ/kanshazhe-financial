## 第一期：核心骨架搭建（2026-06-15）

### 本期工作概要

完成了财务机器人平台的 MVP 骨架搭建，涵盖前后端全链路：

- **后端**：FastAPI + SQLAlchemy + SQLite，5 个数据模型（User/Account/Category/Transaction/Budget），6 组 RESTful API（认证/账户/分类/交易/预算），JWT 认证
- **前端**：Vue 3 + TypeScript + Element Plus + Pinia + Axios，7 个页面（登录/注册/仪表盘/账户管理/交易流水/分类管理/预算管理）
- 前后端通过 Vite 代理联通，一起启动即可使用

### 关键决策记录

| 决策 | 选项 | 选择理由 |
|------|------|---------|
| 后端框架 | Flask vs FastAPI | FastAPI：自动API文档、Pydantic验证、异步支持 |
| 数据库 | SQLite vs PostgreSQL | SQLite：开发阶段零配置，后期可迁移 |
| 前端状态管理 | Pinia vs Vuex | Pinia：更轻量、TypeScript支持更好 |
| 交易类型 | 收入/支出/转账 | 三种基本类型覆盖个人记账核心需求 |
| 转账实现 | to_account_id 字段 | 双账户一增一减，总资产不变 |

### 踩坑记录

1. **FastAPI 路由尾斜杠导致 401**
   - 问题：`@router.get("/")` 对不带斜杠的请求返回 307 重定向，浏览器跟进时丢失 Authorization 头
   - 原因：浏览器安全策略，跨源重定向会剥离认证头
   - 解决：统一用 `@router.get("")` 不要尾斜杠
   - 避免：所有路由定义记得检查有无尾斜杠

2. **Windows 环境下杀 Python 进程**
   - 问题：Git Bash 的 `kill` 命令对 Windows Python 进程无效
   - 原因：Git Bash 的进程管理权限不足
   - 解决：用 Python 的 `subprocess.run(['taskkill', '/F', '/PID', pid])` 或 CMD 的 `taskkill /F /PID`
   - 避免：不要依赖 Git Bash 的 kill，用 `netstat -ano | findstr :端口` 查 PID 后用 taskkill

3. **两个外键指向同一表**
   - 问题：Transaction 同时有 `account_id` 和 `to_account_id` 指向 accounts
   - 原因：SQLAlchemy 无法自动判定用哪个外键建立关系
   - 解决：关系定义加 `foreign_keys="Transaction.account_id"`
   - 避免：多个FK指向同一表时，永远显式写 foreign_keys

4. **Promise.all 一个失败全不显示**
   - 问题：Dashboard 同时请求多个 API，一个失败导致全部数据不加载
   - 原因：Promise.all 的 fail-fast 机制
   - 解决：改用独立 try/catch 或 Promise.allSettled 逐个处理
   - 避免：UI 数据加载不要用 Promise.all，用独立 try/catch

5. **page_size 超限导致 422**
   - 问题：前端传 page_size=1000，后端限制 le=100
   - 原因：后端 Query 参数校验严格
   - 解决：后端改大上限到 10000，或前端别传超限值
   - 避免：对前端批量请求保持后端参数松弛

6. **静默模式吞错误**
   - 问题：全局 silent mode 导致所有错误不可见，用户无法排查
   - 原因：Axios 拦截器吞了错误信息
   - 解决：删除全局静默模式，各组件本地处理错误
   - 避免：不要全局吞错误，让问题可见才能定位

### 有效模式

- **先后端再前端**的开发顺序：先把 API 全部调通（验证工具 curl/TestClient），确认无误后再写前端页面，避免两边同时出问题无从排查
- **Python TestClient** 比 curl 好用的多：可以直接测试 API 逻辑，不受网络端口/进程冲突影响
- **错误需要可见**：静默吞错误看似友好，实际让调试极其困难。错误弹窗虽然烦人，但能帮开发阶段快速定位问题
- **分页参数设大不设小**：对有批量查询需求的接口（交易流水、统计数据），把 page_size 上限设大（10000），避免后续功能开发时被限制卡住

### 架构设计说明

**为什么新建 AccountSubject 而不是复用 Category？**
- Category 是为个人记账设计的（income/expense/transfer 三种类型），缺少企业会计所需的完整分类（资产/负债/权益/成本/收入/费用）
- 会计科目需要标准编码体系、严格的树形层级和借贷方向属性
- 两者将长期共存：个人用户继续用 Category，企业用户用 AccountSubject

**为什么科目编码允许自定义？**
- 标准科目编码符合《企业会计准则》规范，但企业可能有内部管理需要的自定义科目
- 编码唯一性校验确保不冲突，系统级科目不可删除（可停用）

**为什么限制三级层级？**
- 中国会计实务中三级科目已能满足绝大多数场景
- 过多层级增加复杂度和查询成本，如需更细粒度可在名称中体现

### 架构设计说明

**凭证状态机设计**
```
草稿(draft) → 已提交(submitted) → 已审核(approved) → 已过账(posted)
                      ↑                                    |
                      └──────── 反过账(unpost) ←────────────┘
```
- 仅草稿状态可修改/删除
- 审核后不可修改，过账后更新科目余额
- 反过账冲销余额后回到已审核状态

**借贷平衡校验**
- 前端实时校验：显示借方合计 = 贷方合计
- 后端再次校验：防止绕过前端直接调用 API
- 不允许一行同时有借和贷

### 踩坑记录

12. **voucher_no 全局唯一导致不同用户冲突**
    - 问题：`voucher_no` 字段设置了 `unique=True` 全局唯一，但 `_generate_voucher_no` 按 `user_id` 查，不同用户生成相同凭证号时报 UNIQUE 约束 500 错误
    - 原因：授权初期字段设计直接设了 `unique=True`，未考虑多用户场景

### 踩坑记录

**22. fetch PDF 下载导致网络错误**
- 问题：前端 `fetch('/api/vouchers/{id}/pdf', { headers: { Authorization } })` 抛出网络错误
- 原因：fetch 处理二进制响应时，浏览器与 Vite 代理之间可能出现流处理差异（具体原因因环境而异）
- 解决：改用 `<a>` 标签 + `?token=` 查询参数，浏览器原生下载，完全不经过 JS 二进制处理
- 避免：文件下载类功能优先用浏览器原生 `<a>` 标签触发，不要用 fetch/axios 手动处理 blob

**23. PDF 文件名含中文导致 HTTP 500**
- 问题：`Content-Disposition: attachment; filename="凭证-记-2026-06-0001.pdf"` 返回 500
- 原因：HTTP header 只能编码 `latin-1`，中文字符不在其范围内
- 解决：文件名改为纯 ASCII（`voucher-记-2026-06-0001.pdf` 中"记"字用 `?` 替代），或使用 RFC 5987 `filename*=UTF-8''...` 编码
- 避免：HTTP header 中的文件名始终用 ASCII-safe 字符

**24. 批量打印逐步下载体验差**
- 问题：原方案 `setTimeout` 逐个调用 `downloadVoucherPdf`，浏览器弹出 N 个下载窗口
- 原因：逐个下载没有原子性，用户需要等待逐个完成
- 解决：改为后端打包 ZIP，前端一次请求一次下载
- 避免：涉及"多个文件的批量操作"优先考虑后端打包（ZIP/tar），而不是前端逐个处理

**25. axios responseType:'blob' 错误处理陷阱**
- 问题：axios 设 `responseType: 'blob'` 时，HTTP 错误响应体也是 Blob 而非 JSON
- 原因：`responseType: 'blob'` 让 axios 把所有响应（包括 4xx/5xx）解析为 Blob
- 解决：catch 块中检测 `err.response.data instanceof Blob`，用 `await data.text()` 读取后 JSON.parse 提取 detail
- 避免：blob 模式下的错误处理需要手动判断并解析

### 有效模式

1. **后端打包 > 前端逐个处理**：批量文件导出时，后端打包 ZIP 比前端逐个下载更可靠、用户体验更好
2. **提取共用辅助函数**：`_voucher_to_pdf_data()` 被单张 PDF 和批量 ZIP 两个端点复用，避免代码重复
3. **`<a>` 标签 > fetch 下载二进制文件**：浏览器原生 `<a>` 标签 + `?token=` 认证方式比 fetch 处理 blob 更稳定
