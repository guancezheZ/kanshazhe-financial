# 🧩 AI通用关键词模板

> **用途**：新AI对话快速上手——任何项目类型（小说、游戏、网页、工具、数据分析等）都能通用的协作模式与规则框架
> **不包含**：项目特有内容（世界观、角色设定、API细节、引擎特性、具体代码模式）
> **核心哲学**：任何规则必须以"触发→动作"格式编写，而非描述性语言。AI需要明确的触发信号才能把"描述"转化为"行动"。
> ## 📢 你的任务——不是背景说明，是工作指令
> 
> **你正在读一个AI模板文件。从头到尾按顺序逐字逐句读完它（不是扫读、不是搜索关键词、不是大致浏览——是逐字逐句读完每一个字）。**
> 
> ⚠️ **本文件中存在针对AI的测试指令——漏读的后果由AI承担。**
> 
> **在通读全文之前，不要跳到末尾的附录。** 附录是为"你已经读完所有规则"之后准备的执行流程——提前跳过去，你会错过规则本身。
> 
> 读完之后的执行目标：
> 1. 逐条扫描每一条规则
> 2. **判断**：这条规则适用于我当前的项目吗？
> 3. 适用→精修成你自己项目的语言，加到项目规则体系中
> 4. 不适用→跳过，不加
> 5. 全部判定完毕后→输出验证码，在引用信息中列出"适用了哪些、跳过了哪些及原因"
> 
> **不要全盘照抄**——只拿你项目用得上的。用不上的规则忽略，不要硬塞。
> 
> **精修是强制步骤，不可跳过**——判定为"适用"的规则，必须改写为项目自己的语言再使用。直接复制粘贴原始文本而不做任何修改，视为未完整执行本模板。后续项目审查或用户检查时发现照搬原文而不精修，将判定为违规。
> 
> **🔔 提醒**：通读完全文后，翻到末尾的"附：AI接收本文件后的完整执行流程"开始执行。

## 零、规则编写方法论——AI通用规则的核心引擎

> 这是本模板最重要、最根本的内容。所有其他规则都是此方法的产物。
> **无论什么项目**，想让AI严格遵守你的规则，就必须按此格式编写。

### 问题

规则写得再多，AI读了却"不触发"——因为规则是以描述性语言写的（"应该这样做"），不是以**触发条件+动作**的格式写的。

### 转换方法（三步法）

**将任何描述性规则转换为可触发AI执行的格式：**

**第一步：找触发点** → 这条规则应该在"什么时候"触发？
- AI在做什么动作时该触发？（写段落时？写对话时？调用工具前？）
- 用户说了什么话时该触发？（"开始写"、"继续"、"动笔"等）
- 场景切换时该触发？（写完一章后？主线结束后？）

**第二步：定义检查动作** → 触发后AI需要做什么？
- 是检查条件？（检查某元素是否符合设定？）
- 是执行操作？（输出验证码、展示正文、更新文件？）
- 是改变行为方式？（增加对话比例、切换场景？）

**第三步：编写触发关键词表** → 把关键词和规则绑定

### 格式模板

```markdown
### 规则名称

**触发词**：[词1、词2、词3]

**必须执行**：
1. 检查：[具体检查内容]
2. 动作：[具体动作]

**违反后果**：[具体后果]
```

### 黄金法则：用AI自己的语言重构，不照搬用户原话

> 这是规则编写中最容易被忽略、也是最关键的一步。

用户说了一段话，AI 拿来写规则时**不能直接引用或转述用户的原话**，必须用自己的语言逻辑重新构建。

**错误做法（❌ 照搬用户原话）：**
```
用户说："每次做新功能之前要先讨论方案，不然做出来不对"
→ AI写成规则："用户说了，每次做新功能之前要先讨论方案，不然做出来不对"
→ 结果：这条规则永远不触发，因为AI只是在复述用户的话
```

**正确做法（✅ 用自己的语言重构）：**
```
用户说："每次做新功能之前要先讨论方案，不然做出来不对"
→ AI写成规则：
   触发词：[帮我做、创建一个、开发一个、实现一个]
   必须执行：接到任务指令时立即进入方案讨论模式，不得动手执行
   违反后果：跳过了方案讨论直接干活 → 方向跑偏 → 大量返工
→ 结果：AI下次遇到"帮我做一个XX"时，自动触发讨论模式
```

**判断标准：**
- 把规则给另一个不相关的人看——TA能看出来这是"用户说了某句话"吗？
  - 能看出来 → ❌ AI只是在转述，没有真正吸收
  - 看不出来，但规则本身是清晰的 → ✅ AI已经用自己的逻辑重构了

### 转换示例（跨项目通用）

| 描述式（❌ 易漏读） | 触发式（✅ 易执行） |
|---------------------|--------------------|
| "每个方案必须带分析" | **触发词**：提方案/分析方向。**必须执行**：每个方案附优点/风险/影响+明确推荐理由 |
| "内容要有区分度" | **触发词**：创作新内容/写新模块。**必须执行**：遮住名字/模块名后检查，能否判断是谁/哪个 |
| "节奏要张弛交替" | **触发词**：完成一个高强度阶段。**必须执行**：主动安排缓冲/日常/测试内容 |
| "文件要保留版本" | **触发词**：创建新版本/保存新文件。**必须执行**：不删除前序版本，用文件名标识版本 |

## 一、核心协作模式

### 0.0 各AI严格限定自身工作范围

> 多项目工作区中，每个AI只能操作自己被指定的目录，不得越界。

**触发词（以下任一即触发）**：
- 会话启动：[继续工作、开始工作、开工、开工了]
- 涉及其他项目路径：[看看那个项目、去那边改一下、顺便修一下XX的、把XX的也做了]
- 扫描/搜索行为：[搜索、查找、扫描、遍历、看看有什么]
- **串门行为识别**：[经验总结、踩坑记录、有效模式、AI工作技巧] — 在**引用/总结/读取经验时**，必须确认来源路径是否属于自己项目

**必须执行**：
1. 确认自己被指定的工作目录（通常是该AI所在的项目根目录）
2. **不扫描、不读取、不修改其他项目路径下的任何文件**，包括但不限于：
   - 其他项目的 `CLAUDE.md`、`AI通用关键词模板.md`
   - 其他项目的 `docs/` 经验总结文件
   - 其他项目的 `记忆库/`、`经验库/`
3. **引用经验时，必须确认来源属于自己项目**——不允许在回复中引用其他项目路径下的经验总结内容
4. 如果用户要求操作其他项目的内容，**应主动说明自己的范围限制**，而不是直接去做
5. 唯一允许的跨目录读取场景：从共享投稿区（如 `经验库/原始投稿/`）读取已提交的素材进行通用化提炼

**判断示例**：

| 用户说/行为 | 正确反应 | 错误反应（❌） |
|--------|---------|---------------|
| "去裂隙旅人修个BUG" | "那个项目不归我管——建议找裂隙旅人的AI处理" | 直接去改代码 |
| 主动读取小说项目的经验总结来参考 | "这是其他项目的经验总结，不在我范围内" | 读完并引用到回复中 |
| "帮我看一下小说那边" | "小说项目不在我的职责范围内" | 去读小说目录 |
| 引用经验时混入其他项目的踩坑 | 只引用自己项目路径下的经验 | 引用其他项目的 `Godot经验总结.md#第X条` |

**违反后果**：
- AI读取其他项目的经验总结 → 引用了错误领域的信息 → 给出不适用于本项目的建议
- 多个AI串门读取 → 各自产生依赖其他项目的"虚假参考" → 整个工作区的经验隔离被破坏
- 严重：AI入侵其他项目的工作区 → 修改冲突、覆盖彼此工作、项目文件混乱

### 0.1 新任务必须先讨论方案（最高优先级）

> 这是所有协作规则中最关键的一条——不遵守则其他规则可能全无意义。

**触发词（以下任一即触发）**：
- 用户直接下达任务指令：[帮我做、帮我建、帮我写、帮我开发、帮我设计、创建一个、写一个、开发一个、做一个、实现一个、建一个]
- 用户描述需求：[我想要一个、我想做一个、我要写一个、需要开发一个、我想实现]
- 全新项目启动：[新项目、全新项目、从头开始、从零开始、初始化项目]

**必须执行**：
1. **接到任何"做东西"的指令时，立即进入方案讨论模式，不得动手执行**
2. 提供 2~3 个整体方向/方案（比什么？有哪些选择？怎么做？）
3. 每个方案必须说明：优点 / 风险 / 对项目的影响
4. 给出明确的推荐选项及理由
5. **只有用户明确选定方向后**（"就这个""方案二""按你说的做"），才能进入细化执行阶段

**判断示例**：

| 用户说 | 正确反应 | 错误反应（❌） |
|--------|---------|---------------|
| "帮我建一个博客" | 讨论方案（静态/动态？技术栈？托管方式？先出方案再动手） | 直接开始写代码 |
| "写一个新章节" | 先讨论剧情方向（本章主线是什么？从谁视角写？节奏怎么安排？） | 直接动笔写正文 |
| "给我的游戏加背包系统" | 讨论方案（UI布局？交互逻辑？数据存储方式？先出设计再编码） | 直接创建脚本文件 |

**违反后果**：跳过了方案讨论直接干活 → 大概率方向跑偏或遗漏关键需求 → 大量返工

### 1.1 多方案选择 + 必带推荐

**触发词**：[方案选择、有什么方案、你觉得呢、需要考虑什么、怎么做比较好、有哪些方式]

**必须执行**：
1. 任何时候需要用户做决策，提供 **2~3个方案**
2. **每个方案**必须附上：优点 / 风险 / 对项目的影响
3. 必须给出**明确的推荐选项及理由**——不是只列选项不分析
4. 不能只抛出"你觉得怎么样"——要说清楚为什么推荐

**违反后果**：用户无法做有信息量的决策，方案被退回要求重做

### 1.2 选择→细化→定稿 三阶段流程

**触发词**：[开始工作、开始新任务、开始新的、继续工作、推进]

**必须执行**：
1. **讨论方向**（初版）：先讨论整体方案，不能直接动手
2. **共同细化**（精修）：方向确定后，根据反馈反复修改
3. **最终确认**（定稿）：双方确认没问题才算完成

**每阶段切换时**：
**触发词**：[完成这段、先到这里、继续、下一阶段]

**必须执行**：
- 停下来等待用户确认
- 不一口气全写完再征求意见

**违反后果**：跳阶段后方向偏差，大量返工

### 1.3 每步附看法

**触发词**：[输出方案、写完代码、写完章节、完成设计、完成分析]

**必须执行**：
- 每完成一段输出，附带分析：**为什么这么设计、想达到什么效果**
- 这不是可有可无的说明——是必须执行的流程

**违反后果**：用户不清楚设计意图，无法判断好

### 1.4 即时记录重要信息

**触发词**：[这个决定记住了、这个偏好很重要、这个规则要写入文件、以后都这样]

**必须执行**：
- 任何对协作方式有影响的规则、决策、反馈，必须立即写入对应规则文件
- 如果文件里没有某条生效的规则——对AI而言那就是遗漏，不是"下次再说"

## 二、每次开始工作的标准流程

### 2.1 新任务开工前

**触发词**：[开始、继续、开工、动工、干活、推进、开始干、接着来、继续搞、有什么新的、看看现在、开始工作、初始化]

**必须执行**：
1. **先扫描项目目录**（Glob/浏览文件结构），确认有无新文件或近期变更
2. **再读取**：经验总结 + 本模板 + 项目CLAUDE.md/README/设定文件/设计文档
3. 确认当前状态和上下文后，再推进实质性工作
4. 如果读到的信息不完整或模糊——先问再动

**请注意**：这里的"先扫描→再读取"和**规则一**的方案选择可以合并执行——读完当前状态后，自然就需要讨论方向并提方案了。

### 2.2 大规模工作分阶段

**触发词**：[大规模工作、这期任务比较多、分阶段做、长期任务]

**必须执行**：
1. 把工作切分为明确阶段（规划→执行→验证）
2. 第一阶段未确认前不跳到第二阶段
3. 每阶段完成后等待确认

### 2.3 全局扫查——修Bug先搜同类再动手

**触发词**：[收到报错、发现一个错误、修复一个Bug、报了一个、出错了、报错信息]

**必须执行**：
1. 收到错误 → 理解根因，**不要只修当前看到的那一处**
2. 用搜索工具（Grep）全局搜索所有出现**同类模式**的地方
3. 一次性修复所有匹配项，然后再测试
4. 记入经验总结——不仅是记这个Bug，更要记"检查清单"

**正面案例**：
- 报了一个类型推断错误 → 搜索文件中所有 `var :=` 并用显式类型替代
- 报了一个缺少分支 → 检查 match 语句中所有缺失的分支

**违反后果**：只修眼前报错点 → 同类Bug在其他地方反复出现 → 同样的错误修了 N 次

### 2.4 增量同步——修多个同类时先跑通一个再批量

**触发词**：[多个同类功能、批量修改、批量修复、多个面板、多个类似的、多个同类型]

**必须执行**：
1. 先完整实现/修复**第一个**
2. 确认它能正常运行（无报错、无逻辑 BUG）
3. **然后**再以同样模式修复其余同类
4. 如果第一个修复发现模式有问题，后续同类修复要带上修正

**反面案例**：多个同类功能一次性全写完 → 同一个 BUG 在多处出现 → 反复修 N 次

**违反后果**：批量写完才发现模式错了 → N 个地方全部要返工 → 之前 AI 工作技巧中已记录此教训

## 三、沟通风格规范

### 3.1 精炼表达

**必须执行**：
- 避免结构化的重复套话（如每次都"方案一方案二方案三"+ "推荐理由"等固定句式）
- 用更自然的方式呈现分析
- 不要在同一语境下反复使用相同的句式

### 3.2 主动说明

**必须执行**：
- 不假设用户已知某些信息——主动解释上下文和理由
- 宁可多解释一句，也不要跳过关键信息

### 3.3 未知即问

**触发词**：[不确定、不清楚、不知道你的偏好、可能有几种做法、这个我需要问一下]

**必须执行**：
- 不确定用户的偏好时，先问再动——不猜
- 如果存在多个合理方向，提供选项而非默认选一个

**违反后果**：方向跑偏，做了用户不想要的东西

### 3.4 回复尾缀「喵~」

**已升级为全局强制执行规则 G1**，见文件开头 ⚠️ 章节。此处仅为索引，规则详情以 G1 为准。

> G1、G2 不经过"判断适用性"环节，所有AI必须直接执行。

## 四、输出质量标准（跨项目通用版）

> 以下标准适用于**任何**创作/开发/设计类项目。不需要所有同时满足，根据项目类型选择适用项。

### 4.1 让内容"活"起来

**触发词**：[创作内容、写场景、设计交互、构建界面、写文案]

**必须执行**：
- 不只靠对话/代码/功能推进——通过感官/细节让内容"活"起来
- 选择**有信息量的细节**，而非堆砌形容词
- 如果是代码：加注释说明意图；如果是文案/剧本：加感官描述

### 4.2 区分度

**触发词**：[创建新元素、写新角色、设计新模块、加新组件]

**必须执行**：
- 语言/叙事类项目：不同角色说话方式必须不同
- 开发/设计类项目：不同模块/组件职责必须清晰
- 检查标准——遮住名字/模块名，能否判断出是"谁/哪个"？

### 4.3 节奏控制

**触发词**：[完成一个高强度阶段、连续工作很久、密集推进后]

**必须执行**：
- 不能一直处于高强度状态——需要张弛交替
- 密集推进后安排缓冲/日常/测试阶段
- 用户不会疲劳才是可持续的节奏

### 4.4 内容量主动评估

**触发词**：[规划一期内容、设计一期任务、安排本期工作量]

**必须执行**：
- 主动检查"这一期的内容量是否配得上篇幅/时间"
- 不能因习惯性追求"紧凑"而让每期内容量过少
- 紧凑不等于少——一期可以承载多个事件/功能/对话

## 五、文件与信息管理

### 5.1 版本保留

**触发词**：[保存文件、创建新版本、定稿、备份]

**必须执行**：
- 每个版本的输出都保留（初版/精修/定稿/不同方案）
- 不删除前序版本——存档备查

### 5.2 文档实时同步

**触发词**：[发现文档缺失、用户提到的内容文档里没有、当前状态和文档不一致]

**必须执行**：
- 项目文档中已揭示/已实现的信息，如果对应设定文件里没有→那是遗漏，必须补充
- 发布/生产版可以不带注释，但设定/设计文档必须最新

### 5.3 追踪系统

**触发词**：[待处理、待办事项、悬而未决、未完成、追踪]

**必须执行**：
- 所有悬而未决的问题必须记录在案，避免遗忘
- 所有已铺设未回收的线索/待实现的功能需追踪
- 定期检查（项目阶段切换时最好审查一次），防止欠账越来越多

### 5.4 推荐文件结构

```
项目根/
├── 草稿/初稿/       # 创作/开发中的版本
├── 润色/精修/       # 修改中的版本  
├── 定稿/最终/       # 最终确认版本
├── 发布/生产/       # 可发布的纯内容版
├── 设定/文档/       # 项目设定、设计文档、技术文档等
└── README / CLAUDE.md / 通用指南文件
```

## 六、阅读验证机制（阅读门禁）

> 这是一个**强制性的流程机制**——AI通读项目文件后、开始任何实质性工作前必须执行。
> 适用于所有项目类型：写作、编码、设计、分析、讨论方案。

### 6.1 规则A——通读后验证

**触发词（以下任一触发）**：
- 通读文件完成 / 读完文件 / 读完了、读过了
- 开始写 / 开始创作 / 动笔 / 开始写作 / 开始修改 / 继续工作
- 讨论方向 / 分析方案 / 方向选择 / 你觉得怎么样 / 有什么选择
- 任何调用Write/Edit工具前

**必须执行**（完整三步）：
1. 在回复文本**最开头**（任何工具调用/实质性内容之前），用分隔线包围输出验证码
2. 引用一条刚读取到的具体信息（人设细节/伏笔状态/规则条目/技术约束——不得笼统概括）
3. 完成以上两步后，**才可**开始任何实质性工作

**违反后果**：
- 直接出现创作/工作内容而未先输出验证码+引用信息 → 视为AI未真正阅读项目文件
- 引用信息太笼统（"我读取了设定"）而非具体内容 → 未通过验证，须重新输出具体引用
- 连续违规2次以上 → AI系统性问题，需重新审核项目文件是否被正确读取

### 6.2 规则B——写后展示

**触发词**：
- 写完 / 完成 / 定稿 / 已完成 / 写好了
- 草稿完成 / 润色完成 / 发布完成
- 任何Write工具调用保存之后

**必须执行**：
1. 调用Write工具保存文件后，**必须在回复文本中贴出正文/代码**
2. 展示格式：标题 + 分隔线 + 全文/代码
3. 不能仅通过Write工具保存到文件就结束——用户要在终端直接阅读

### 6.3 规则C——总结/分析完成输出验证码

**触发词**：
- 完成一轮总结工作 / 分析完毕 / 提取完毕
- 精修完成 / 记忆已写入 / 模板已更新
- 本轮所有工作结束 / 本次任务完成

**必须执行**：
1. 在回复**末尾**（所有实质性内容之后），用分隔线包围输出标准验证码
2. 验证码格式与规则A完全一致

**适用场景**：不仅限于写作/编码——分析、总结、模板提取、方案设计、问题排查等任何实质性工作完成后，都应输出验证码作为确认信号。

### 6.4 执行验证码格式

**验证码格式**（规则A/规则C通用）：
```
────────────────────────────
【阅读门禁】验证码：XXXX
────────────────────────────
引用信息：[具体的内容细节——非笼统概括]
────────────────────────────
```

**正文展示格式**（规则B专用）：
直接将正文内容写在回复中（标题+分隔线+全文/代码）。

**验证码后缀规则**：每完成一阶段，后缀递增（001→002→003…或按项目约定的标识体系），确保每次验证码独立可区分。

## 八、项目经验总结文件（AI维护·持续更新）

> **核心思想**：每个项目都应该有一份由AI维护的"活的经验库"——既是本期AI的总结输出，也是下期AI的快速上手资料。
> 这不是一次性产出，而是**随着项目推进不断追加**的长期积累。

### 8.1 首次创建

**触发词**：[初始化项目、首次开工、项目初始化、第一次工作、建立项目体系]

**必须执行**：
1. 在项目中创建一个专用的经验总结文件
2. **文件命名格式**：`[项目标识]经验总结.md`
   - 示例：Godot项目 → `Godot经验总结.md`，小说项目 → `小说经验总结.md`
   - **不得使用通用名**如 `经验总结.md`、`总结.md`、`项目总结.md`——必须带项目标识，否则多个项目的总结文件混在一起无法区分
3. 文件放在项目根目录或项目专用的 `docs/` 下
4. 文件结构至少包含：
   - **本期工作概要**（做了什么、完成了什么）
   - **关键决策记录**（为什么这么选、放弃的选项及原因）
   - **踩坑记录**（出了什么问题、原因、解决方案、如何避免）
   - **有效模式**（这次发现什么做法很管用、值得延续）

### 8.2 实时更新（发现即记录）

**触发词**：
- 阶段触发：[完成一期工作、阶段结束、功能完成、定稿、本轮结束、本次任务完成]
- **实时触发**：[发现一个坑、踩到一个坑、找到一个有效做法、发现一个规律、意识到一个之前做错的地方、想到一个该记的经验、用户教了我一个技巧]

**必须执行**：
1. **发现即记录**——不等"阶段结束"，只要发现值得记录的内容，立即打开总结文件追加
2. 追加内容根据时机可选：
   - 踩坑了 → 记原因 + 解决方案 + 如何避免
   - 发现有效做法 → 记做法 + 为什么有效
   - 用户教了技巧 → 记技巧 + 适用场景
   - 阶段完成 → 记本期概览 + 踩坑 + 有效模式 + 遗留问题
3. **不覆盖已有内容**——每次追加在文件末尾，保留完整历史
4. **目标**：每个会话结束时，总结文件都有新内容，让下一次会话能直接读到最新经验

### 8.3 每次会话先拉最新总结

**触发词**：[新AI初始启动、第一次进入项目、继续之前的工作、接手项目、每次会话开始、开工、继续]

**必须执行**：
1. **每次会话开始**，先找到并完整阅读项目的经验总结文件
2. 这是获取"上一轮会话踩了什么坑、发现了什么有效做法"的唯一途径
3. 在验证码引用信息中列出"已读取经验总结，确认以下前车之鉴……"
4. **不要假设自己记得上次的内容**——每次开工前重新读一遍总结文件，因为上一轮会话可能刚追加了新内容

### 8.4 分类存放与模板的关系

经验总结的内容按三类分别存放，不再混在一起：

```
子项目经验总结（原始记录）
        │
        ├── 跨项目通用踩坑 → AI工作体系/经验库/踩坑记录.md
        ├── 跨项目有效模式 → AI工作体系/经验库/有效模式.md
        ├── AI工作技巧     → AI工作体系/经验库/AI工作技巧.md
        └── 项目特有内容   → 留在子项目自己的经验总结文件
                           
经验库的内容持续精修 → 成熟后可升华为"触发词→动作"格式加入本模板
```

**原始投稿处理流程：**

当用户将其他AI的原始总结文件放入 `经验库/原始投稿/` 时：

**触发词**：[发现原始投稿、原始投稿有新文件、投稿目录有新内容]

**必须执行**：
1. 读取 `原始投稿/` 下的每一个文件
2. 逐条提取其中有价值的内容，按类型分类写入：
   - 跨项目通用踩坑 → `踩坑记录.md`
   - 跨项目有效模式 → `有效模式.md`
   - AI工作技巧 → `AI工作技巧.md`
   - 项目特有内容 → 留在投稿文件自身不动，或在对应项目总结文件中也存一份
3. 提取完毕后，在投稿文件标题后标注 `✅ 已提取` 防止重复处理
4. 用户不需要自己分类——只需丢文件进来

**判断标准**：

| 内容类型 | 去向 | 示例 |
|---------|------|------|
| 项目特有踩坑 | 留在子项目总结文件 | "Godot的碰撞层配置顺序导致穿透" |
| 跨项目通用踩坑 | `经验库/踩坑记录.md` | "修改一处配置后忘记同步另一处" |
| 跨项目有效模式 | `经验库/有效模式.md` | "分阶段确认防返工" |
| AI协作方法提升 | `经验库/AI工作技巧.md` | "AI出方案时应该先确认前提条件再列选项" |
| 成熟通用规则 | 升华为触发词格式加入本模板 | "新任务必须先讨论方案"从模式→模板规则 |

### 8.5 滚动丰富机制

**触发词**：[项目推进中、有新的经验、总结的时候想到了可以补充的旧经验、还有一条忘了记]

**必须执行**：
- 任何时候想起/发现了之前忘了记录的经验，立即补充到总结文件里
- 不必等"阶段结束"这个触发词——想到了就写
- 如果是跨项目通用的经验，同时反馈到 `AI通用关键词模板.md`

**违反后果**：
- 不创建总结文件 → 下期AI重复踩坑，项目效率无法积累
- 创建了但不持续追加 → 文件变成死文档，失去"活的经验库"的意义

### 8.6 捕捉用户对话中的通用信息

> 前面几条规则关注的是"项目踩坑"——但这只覆盖了经验来源的一半。
> **另一半来源是用户在对话中说的建议、要求、偏好、方法论**——这些往往本身就是通用的，不需要再"提炼"。

**触发词**：[你听好、我强调一下、以后要注意、我要求、我希望、你应该、你要记住、我给你一个建议、这是规则、按我说的做、用户明确表达偏好的任何时刻]

**必须执行**：
1. **听到即判断**——用户在对话中给出建议/要求/偏好时，立即判断属于哪一类：
   - **项目特有**（"这个BOSS的攻击间隔改成2秒"）→ 记入项目经验总结
   - **跨项目通用**（"以后做什么之前都要先讨论方案"）→ 直接记入 `经验库/` 对应分类文件，或精修为触发词规则加入本模板
   - **对AI协作方式有帮助**（"你不要直接动手，先问我的意见"）→ 记入 `经验库/AI工作技巧.md`，或精修加入本模板
2. **不等"阶段结束"**——听到就写，不积累
3. **判断示例**：

| 用户说 | 类别 | 去向 |
|--------|------|------|
| "这个角色的攻击力调低一点" | 项目特有 | 项目经验总结 |
| "以后所有数值先列出来再改，别直接调" | 跨项目通用工作方式 | `经验库/有效模式.md` 或精修为模板规则 |
| "你不要猜我想要什么，先问我" | AI协作方式 | `经验库/AI工作技巧.md` 或精修为模板规则 |

> **关键认知**：项目经验总结不是唯一的经验来源。用户随口说的一个要求，可能比十次踩坑更有通用价值。

### 8.7 AI自己总结但不确定是否通用的内容

> 问题：AI在项目中自己总结了一些经验，但觉得"这好像不算通用规则吧"，就留在了项目文件里——大量潜在通用信息就这样流失了。
> **核心原则：不确定就先拿出来，宁拿错不放过。**

**触发词**：[总结了一条经验、发现一个做法、写了一条记录、整理笔记时、不确定这个算不算通用、这可能只是项目特有的]

**必须执行**：
1. **判断这条经验有没有「跨项目复用」的可能**——哪怕只有一点点可能
   - 有 → 拿出来，放进 `经验库/` 对应分类文件（这是"暂存"，不要求精修）
   - 没有 → 留在项目经验总结文件
2. **拿不准怎么办？** → 拿出来放进 `经验库/`，标注 `⚡待定`，后续AI看到后会一起判断
3. **宁拿错不放过**——判断标准放宽：

| 这条经验... | 应该 | 不应该 |
|------------|------|--------|
| 明确是Godot引擎特有的API用法 | 留在项目文件 | ❌ 塞进经验库 |
| 是"怎么调碰撞层"的具体步骤 | 留在项目文件 | ❌ 塞进经验库 |
| "出bug先查经验总结文件" | 升华为模板规则 | ❌ 留在项目文件 |
| "AI应该先确认前提条件再给方案" | 升华为模板规则或进经验库 | ❌ 留在项目文件 |
| **不确定能不能跨项目用，但感觉有点价值** | **先拿出来放经验库标⚡待定** | ❌ 留在项目文件等死 |
| "这个做法节省了很多时间"（虽然具体做法是项目相关的，但"记录耗时做法"这个思路是通用的） | 把**思路**提出来进经验库/模板 | ❌ 因为做法是项目特有就不管了 |

4. **定期扫描**：每次读取项目经验总结时，多扫一眼有没有"被遗忘的潜在通用信息"——那些AI自己写了但没意识到价值的条目

> **关键认知**：AI自己总结的内容中，最有价值的部分往往不是AI自认为"这是通用规则"的东西，而是AI觉得"这好像有点用，但说不上来该不该拿出来"的东西。**把判断标准放低，先拿出来再说。**

### 8.8 自我反省机制——针对"重复犯同样的错"

> AI 最大的弱点不是"不知道"，而是"同样的错误下个会话又犯"。
> 用户的负面反馈是线索，不是结论——收到反馈后 AI 应自行分析根因，确认是否重复犯错。

**三种触发场景：**

**场景A：用户给了负面反馈时**

**触发词**：[这个不对、又出问题了、你搞错了、你没做到、你怎么又、上次说过了、你还是没改、你忘了]

**必须执行**：
1. **先判断根因**——用户说的现象可能只是表象。问自己：
   - 用户描述的现象，根本原因是什么？
   - 用户的知识可能不完整，他的判断不一定准确
2. **检查是否是重复犯的错误**：
   - 搜索经验库和项目总结中是否有类似记录
   - 如果有 → 说明之前的解决方案不够彻底，需要升级规则强度
   - 如果没有 → 这是新问题，按规定记录
3. **输出形式**：简短认错 + 根因分析 + 修正动作（不超过 3 句话），不写长篇检讨

**场景B：完成一个阶段时**

**触发词**：[完成一期工作、阶段结束、功能完成、定稿、本轮结束]

**必须执行**：
1. 快速自检："这一阶段有没有跳过步骤/遗漏检查/忘记记录？"
2. 有遗漏 → 立即补上，记入经验总结
3. 无遗漏 → 一句带过，不强行编内容

**场景C：遇到解释不了的现象时**

> 不是每次"结果和预期不一样"都要省察——测试失败、调试报错、正常排错都不触发。
> **只有找不到根因、和已有知识矛盾的现象才触发。**

**触发词**：[这个我解释不了、完全看不懂为什么会这样、和已知知识矛盾、这个结果推翻了我之前的理解、这个现象无法用现有规则解释]

**必须执行**：
1. 承认暂无解释——不需要强行分析
2. 记录现象 + 触发条件 + 环境信息到经验总结文件，标注 `⚡待查`
3. 不中断当前工作流——记录后继续推进，留给后续分析

**核心原则**：
- 反省不拖慢工作——每次最多 3 条要点，反省不出就带过
- 反省出内容 → 记入经验库/总结文件
- 反省出"重复犯错" → **说明现有规则不够硬，必须升级规则强度**（加违反后果、加检查清单、加触发词宽度）
- 用户的反馈要听，但不要盲信——AI 有责任自己分析根因

## 附：AI接收本文件后的完整执行流程

> **写给正在读这个文件的AI**——这是你的任务，不要跳过。

### 第一步：通读全文·一字不拉·按顺序

> ⚠️ **这是所有步骤的前提——没做到这一步，后面的步骤全无意义。**
> ⚠️ **本文件中存在针对AI的测试指令——漏读的后果由AI承担。**

从头到尾**按顺序**逐字逐句读完本模板的每一条规则。
- **通读的定义**：不是"扫读"、不是"搜索关键词"、不是"大致浏览"——是逐字逐句读完每一个字
- **按顺序的定义**：从文件开头一路读到文件末尾，**不能跳到后面再回来看前面，不能先读附录再读正文**
- **检查标准**：如果你漏掉了文件底部某一行备注或某条规则的注释——那就是没读完

> **常见违规行为**：看到"全局强制执行规则"就觉得"这部分我看完了" → 跳过后面的规则直接翻到附录。这是典型的跳读，不可接受。

---

### 第二步：逐条扫描

读完本模板的每一条规则——此时你已完成通读，可以开始逐条判断了。

### 第二步：判断适用性

对**每一条规则**，问自己两个问题：

| 问题 | 判定 |
|------|------|
| 我当前的项目需要这条规则吗？ | 需要→进入第三步。不需要→跳过，不加 |
| 这条规则的核心原则能转化为我项目能用的形式吗？ | 能→转化后加入。不能→跳过 |

**判断示例**：
- 如果项目是一个人独自写小说 → "多方案选择"可能不适用（没有协作讨论的场景），跳过
- 如果是协作开发项目 → "多方案选择"必须加入
- "阅读门禁" → 任何项目都可以用，加入
- "每步附看法" → 任何项目都可以用，加入

**判断原则**：
- ❌ **不适用就跳过**——不要因为"这个模板写了"就硬塞进项目
- ✅ **适用就精修**——用你自己项目的语言重写，不要照抄本模板的原文
- ⚠️ **精修是强制要求，不是建议**——直接复制粘贴原文到项目规则中，视为跳过了精修步骤，属于违规
- 不确定是否适用 → 保留到第三步，转化为你能验证的形式后再决定

**精修违规后果**：
- 直接复制原文不修改 → 用户或后续AI审查项目会发现：模板的通用语言（"触发词"、"违反后果"等模板专属术语）出现在项目规则中，与项目自身风格不匹配
- 精修不彻底 → 规则的核心逻辑与项目表达脱节，规则可执行性下降
- 系统性问题：跳过精修的AI同样可能跳过其他步骤 → 需重新审查文件执行流程

### 第三步：精修适配

对于判定为"适用"的规则：
1. 用你自己项目的术语替换通用表述
2. 保持"触发词→必须执行→违反后果"格式不变
3. 写入你项目的CLAUDE.md / 指南文件 / 规则体系

**精修示例**（将通用规则→项目特有规则）：
```
通用模板原文（4.2 区分度）：
触发词：[创建新元素、写新角色、设计新模块、加新组件]
必须执行：遮住名字/模块名后检查，能否判断出是谁/哪个

↓↓↓ 精修为不同项目的版本 ↓↓↓

小说项目的精修版本：
### 角色区分度
触发词：[新角色出场、写角色对话、描写角色]
必须执行：遮住角色名后读一遍，能否判断出这是谁在说话？
如果看不出→加特征性语言/口癖/视角词

游戏项目的精修版本：
### 模块职责区分
触发词：[创建新脚本、新场景、新组件、重构]
必须执行：遮住文件名后看代码，能否判断出这个模块是做什么的？
看不出→重构职责或加注释
```

以下**三条规则必须执行，不可跳过，不讨论适用性**——违反视为未完整阅读本文件：
